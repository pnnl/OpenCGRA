# OpenCGRA Simulation Technical Documentation

## 1. Introduction

Coarse-Grained Re-configurable Arrays (CGRAs) are reconfigurable accelerators for spatial computing. They consist of a 2D array of word-level processing elements (PEs) and an on-chip interconnect. Each PE includes basic arithmetic/logic functions and registers, and is connected to neighboring PEs through switches. Both the interconnect and functional units can be reconfigured every cycle. This architecture strikes a balance between flexibility and energy efficiency. Compared with general-purpose processors, CGRAs reduce instruction overhead through pre-scheduled computation and routing, thereby improving energy efficiency. Compared with fine-grained FPGAs, CGRAs can achieve higher performance with lower power under word-level reconfiguration. A CGRA can be configured in both space and time: by programming the on-chip configuration memory, it can reuse a limited number of processing elements across multiple cycles to implement larger computation graphs.

OpenCGRA is an open-source CGRA architecture generation and simulation framework. Through parameterized design and multi-level models, it enables researchers and engineers to rapidly construct CGRAs of different sizes and topologies, and to validate algorithms and hardware implementations at the Functional Level (FL), Cycle-Accurate Level (CL), and Register-Transfer Level (RTL). The framework provides a complete flow from configuration mapping, model construction, and simulation to Verilog generation. It is suitable both for validating new algorithms and for integrating custom kernels and generating synthesizable RTL models.

This technical document is written based on the OpenCGRA simulation flow, and targets novice engineers, researchers, and engineering integrators. The document targets a specific commit version of OpenCGRA; readers should reproduce experiments using the specified version. Interfaces may differ across branches. The referenced repositories, tool versions, and configuration examples are provided in the corresponding sections.

After completing this guide, readers will be able to:

- Install and configure the software environment required by OpenCGRA, and understand dependency/version constraints.
- Understand the roles and differences among the FL, CL, and RTL models, and choose an appropriate level for validation.
- Generate control-word sequences based on `config.json`, `dfg.json`, and data/constant files, and apply them to different model levels.
- Build a test harness, run simulations, collect outputs, and compare them with golden reference results.
- Debug common failure modes (e.g., control-word loading errors, routing configuration errors, handshake stalls, etc.).

## 2. Environment Setup and Installation

OpenCGRA is developed for Python 3.7 and only supports Linux distributions such as Ubuntu 18.04/20.04. Major dependencies include:

- Graphviz — used to draw control and data flow graphs; must be installed via a package manager in advance.
- Verilator (≥ 4.0) — translates Verilog generated by PyMTL3 into a C++ simulation library.
- PyMTL3 — a Python-based hardware modeling and simulation framework; all OpenCGRA model levels are implemented on top of it.
- Other Python packages specified by `requirements.txt`.

3. Install PyMTL3 and dependencies: run `pip install -r requirements.txt` in the virtual environment, and ensure dependencies such as `cffi` are installed correctly.

## 3. OpenCGRA Simulation Flow Overview

OpenCGRA provides three abstraction-level models to represent a CGRA with different accuracy and cost. Understanding the role of each level is critical for correct simulation.

### 3.1 Layering

| Layer Name | Primary Purpose | Input Files/Interfaces | Output Information | Control Memory Implementation | Verification Method |
|---|---|---|---|---|---|
| Functional Level (FL) | Fast functional verification;<br>used as the golden reference;<br>checks algorithm correctness and constant/data dependencies | FuDFG (functional dataflow graph),<br>`DataType`, `PredicateType`, `CtrlType`,<br>constant arrays | Final computation results and<br>updated data memory | No hardware control memory;<br>executes node-by-node in DFG order | Compare against mathematically expected results;<br>used as the golden standard |
| Cycle-Accurate Level (CL) | Simulate data/control flow across multiple tiles;<br>evaluate routing, timing control, and flow-control strategies | `FunctionUnit` list,<br>`FuList`, `DataType`, `PredicateType`, `CtrlType`,<br>array dimensions,<br>`preload_ctrl`, `preload_data`, `preload_const` | Per-cycle `line_trace`,<br>per-tile output ports,<br>and `DataMem` state | Uses `CtrlMemCL` and preloads the operation list at construction | Compare with FL outputs;<br>use traces to check deadlock / II issues |
| Register-Transfer Level (RTL) | Accurate hardware modeling;<br>generate Verilog and synthesize;<br>evaluate handshake protocols and timing violations | `DataType`, `PredicateType`, `CtrlType`,<br>array dimensions,<br>control memory size,<br>control-sequence length,<br>functional unit list,<br>`preload_data`, `preload_const` | Cycle-accurate output-port signals;<br>Verilog/VCD waveforms | Uses `CtrlMemRTL` register file;<br>writes control words one-by-one via `recv_waddr` and `recv_ctrl` ports | Compare with CL/FL outputs;<br>inspect waveforms and handshake status;<br>synthesizable to FPGA/ASIC |

Key cross-layer differences: at the CL level, control memory is written once at construction via `preload_ctrl`, while at the RTL level control words must be written sequentially via ports. If the write order is incorrect, the control flow will become corrupted. The CL model abstracts part of the handshaking and register delays, while the RTL model strictly follows the ready/valid protocol; constant queues and registers are explicitly modeled at the RTL level.

### 3.2 Simulation Steps

The simulation flow includes the following key steps:

1. Dependency check: confirm installation of Python 3.7, Graphviz, Verilator, PyMTL3, etc.; otherwise the model cannot be built or Verilog translation will fail.
2. Start the Python environment and clone the repository: fetch the specified version of OpenCGRA via `git`, and ensure the code structure matches the relative imports used by test scripts.
3. Prepare DFG and configuration files: provide `config.json`, `dfg.json`, and constant/data files, and generate control sequences using CGRA-Mapper or custom scripts.
4. Build the FL model (optional): call `CGRAFL` in `cgra/CGRAFL.py` to produce golden outputs using the DFG as input.
5. Build the CL model: call the `construct` method of `cgra/CGRACL.py`, passing the functional unit list, data type, control type, array dimensions, and `preload_ctrl`, `preload_data`, `preload_const`.
6. Build the RTL model: call the `construct` method of `cgra/CGRARTL.py`, providing array dimensions, control memory size, control sequence length, and the functional unit list; control words are written through `recv_waddr` and `recv_ctrl`.
7. Build and connect the TestHarness: use the `TestHarness` class in `cgra/translate/CGRARTL_test.py` to compose the DUT (CGRACL or CGRARTL), data memory, and control-write interface.
8. Model elaboration: call `test_harness.elaborate()` to elaborate the module hierarchy and establish connectivity.
9. Verilog translation and import (RTL): for hardware-level simulation, set `verilog_translate_import=True`, then call `TranslationImportPass()` to translate the model into Verilog and have Verilator generate a C++ simulation library.
10. Apply `SimulationPass` and reset: apply `SimulationPass()` to the TestHarness, then call `sim_reset()` to initialize state.
11. Cycle-by-cycle ticking: call `tick()` under a `max_cycles` bound, and use `done()` to determine completion; too many cycles may indicate deadlock or configuration errors.
12. Result sampling and assertions: after simulation, read each tile’s output ports and `DataMem` state, and compare them against FL outputs; if mismatched, use `line_trace()` and `ctrl_mem.line_trace()` to locate issues.
13. Optionally generate Verilog/Verilator outputs: for synthesis or FPGA deployment, save the Verilog and the C++ simulation library generated by `TranslationImportPass`.

These steps form the complete workflow from repository setup to simulation validation. This section only summarizes each stage; detailed operations are explained in later sections.

## 4. Input Artifacts and Mapping Flow

OpenCGRA simulation depends on multiple JSON input files and preload artifacts. These artifacts determine the CGRA topology, functional unit configuration, and task mapping.

### 4.1 `config.json`

`config.json` describes the CGRA topology and parameters, such as array width/height, per-tile functional unit list (`FuList`), routing topology, crossbar port count, control memory size, and overlapped iteration interval (II). Key fields include:

- `width` and `height`: number of columns and rows in the CGRA array.
- `ctrl_mem_size`: depth of the control memory, which determines how many control words can be stored.
- `data_mem_size`: capacity of the data memory, used for loop buffers or standalone storage.
- `FuList`: per-tile functional unit list, which determines the function-selection bit-width in `CtrlType`.
- `switch` / `connectivity`: defines crossbar port count and connectivity topology, determining the bit-width of routing types.
- `II`: initiation interval for time-multiplexed execution; it should satisfy loop unrolling requirements, otherwise deadlock may occur.

When parsing `config.json`, pay attention to bit-width consistency. The functional unit list affects the bit-width of the `opt` field in `CtrlType`, and the routing topology determines the bit-width of `RouteType`. Incorrect bit-width settings may cause control words to overflow or be truncated.

### 4.2 `dfg.json`

`dfg.json` stores the dataflow graph generated by CGRA-Mapper or other compilers. Fields include node ID, operation type (`opt`), input/output edges, constant indices, predicate requirements, etc. For each DFG node:

- `opt` corresponds to an arithmetic operation or a branch in the algorithm.
- `in_edges` and `out_edges` define data dependencies.
- `num_const` indicates how many constants must be read from `preload_const`.
- `predicate` and `branch` markers determine control-flow behavior and must be mapped into corresponding fields in `CtrlType`.

When writing a custom DFG, ensure nodes correspond one-to-one with the reference function. In particular, when handling branches and predicates, you must correctly mark the `ive_out_ctrl` and `live_out_val` flags.

### 4.3 Data and Constant Preloading

- `preload_data`: a 2D list or similar array used to initialize each tile’s data memory.
- `preload_const`: per-tile constant queues; each element is a `DataType` value.
- `preload_ctrl` (CL only): a control-memory preload list; each tile must provide the full control-word sequence when constructing the CL model.

These preloads are fixed after model construction and do not change during simulation. When integrating with CGRA-Mapper, you should check whether the generated control-word sequences match the `CtrlType` bit-widths. If necessary, supplement predicate information, resource binding order, and II constraints.

### 4.4 Integrating with CGRA-Mapper

CGRA-Mapper is a compiler that maps high-level algorithms onto a CGRA. When using CGRA-Mapper, you provide the kernel C code and the target CGRA configuration; it generates `config.json`, `dfg.json`, and per-tile control-word sequences. Because the output may lack predicate bits or resource binding order, users may need to fill them in manually and verify that control-word bit-widths match `CtrlType`.

## 5. Detailed Simulation Procedure

This section expands each simulation step with inputs, outputs, key functions, and potential failure points, helping readers understand the underlying mechanisms during practical use.

### 5.1 Dependency Check and Repository Setup

Before starting simulation, confirm the system environment satisfies Python 3.7, Graphviz, Verilator (≥ 4.0), and PyMTL3 dependencies. Missing dependencies will cause model construction failures or Verilog translation errors. Verify Verilator via `verilator --version`. Check Graphviz via `dot -V`. Clone the OpenCGRA repository and `checkout` to the specified commit; ensure `pytest.ini` is in the repository root, otherwise relative imports may fail.

### 5.2 Control Sequence Generation

Use CGRA-Mapper or custom scripts to map `dfg.json` into control-word sequences. This mapping involves computing per-tile operations and routing configurations while considering II, predicate, and branch information. Incorrect mapping will lead to deadlock or incorrect results. The outputs should include:

- `src_opt`: per-tile control-word sequences; the list length equals the control memory depth.
- `ctrl_waddr`: the corresponding write-address sequence, used by the RTL level to write via `recv_waddr`.
- `preload_ctrl`: the CL-level control preload list used during construction.
- `preload_const` and `preload_data`: used as model construction parameters.

### 5.4 Test Harness and Cycle Driving

TestHarness: used to compose the DUT and external interfaces, implementing control-word writes, data I/O, and done detection. At the RTL level, TestHarness includes the `TestSrcRTL` module, which sends `src_opt` and `ctrl_waddr` entries one-by-one to the `recv_ctrl` and `recv_waddr` ports. At the CL level, `preload_ctrl` is loaded into `CtrlMemCL` during construction, so TestHarness only needs to drive the clock and check outputs.

Elaboration: call `test_harness.elaborate()` to elaborate the module hierarchy and build interfaces and connectivity. Omitting this step will cause `SimulationPass` to access null/empty ports.

TranslationImportPass (RTL): if hardware-level simulation or Verilog generation is needed, set `verilog_translate_import=True` before calling `run_sim()`, and use `TranslationImportPass()` to translate the model into Verilog and have Verilator generate a C++ simulation library. Missing Verilator or incompatible versions will cause translation failures.

SimulationPass: after elaboration, apply `SimulationPass()` to create the simulation engine, and call `sim_reset()` to initialize registers and memory. All `tick` loops must occur after `sim_reset()`, otherwise state is uninitialized.

Tick loop and timeout: drive the clock via `tick()`, one call per cycle, until `done()` returns `True` or `max_cycles` is reached. Too small a `max_cycles` may prevent completion; too large may hide deadlocks. Set it reasonably based on pipeline depth and input length.

### 5.5 Result Sampling and Debugging

After simulation completes, read each tile’s output ports and the final contents of `DataMem`. CL/RTL outputs should match the FL golden results. If there is a mismatch:

1) Check control-word loading: confirm the control words stored in `CtrlMemCL` or `CtrlMemRTL` match `src_opt`.

2) Check routing configuration: verify that routing fields in control words match the `RouteType` bit-width, and that routing topology matches `config.json`.

3) Check predicate/branch updates: print each node’s `live_out_ctrl` and `live_out_val` in the FL model. Observe predicate bits in CL/RTL and ensure branch logic is correct.

4) Check data/constant preloads: confirm `preload_data` and `preload_const` lengths match the number of tiles, and that `DataType` bit-width can hold the constants.

5) Adjust sampling time: if pipelines/loops introduce output latency, adjust `max_cycles` or the sampling point.

## 6. Result Validation and Reference Comparison

Validation typically follows this flow:

1) Run the FL model: call `CGRAFL` with the same DFG, data types, and constants to generate golden outputs.

2) Run CL simulation: build `CGRACL`, load `preload_ctrl`, `preload_data`, and `preload_const`, execute the tick loop and record `line_trace`. Compare per-cycle outputs against FL results to confirm routing and flow control correctness.

3) Run RTL simulation: use TestHarness to write control words and simulate; check RTL outputs match CL outputs. If they differ, inspect `ctrl_mem`, `line_trace`, and handshake signals to locate issues.

4) Tune `max_cycles`: the required max cycle count may differ across layers for the same kernel; adjust `max_cycles` according to pipeline depth and loop unrolling to avoid false timeout reports.

5) Record waveforms and traces: for RTL simulation, generate Verilog/VCD waveforms for timing analysis. It is recommended to provide common testbench scripts and output paths in an appendix for long-term maintenance.
